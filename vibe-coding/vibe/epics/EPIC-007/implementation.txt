Feat-004 | contactValidation.js
---------------------------------
File: vibe-coding/src/website/pages/homePage/contactSection/contactValidation.js

No imports needed — pure JS.

  export function validateContactForm({ name, companyName, email, message }) {
    const errors = {}
    if (!name.trim()) {
      errors.name = 'Name is required.'
    }
    if (!email.trim()) {
      errors.email = 'Email is required.'
    } else if (!/^[^\s@]+@[^\s@]+\.[^\s@]+$/.test(email)) {
      errors.email = 'Please enter a valid email address.'
    }
    if (!message.trim()) {
      errors.message = 'Message is required.'
    }
    return errors
  }

companyName is intentionally not validated — it is optional.

Feat-005 | contactForm.jsx
----------------------------
Files:
  vibe-coding/src/website/pages/homePage/contactSection/contactForm.jsx
  vibe-coding/src/website/pages/homePage/contactSection/contactForm.scss

Imports:
  import { useState } from 'react'
  import emailjs from '@emailjs/browser'
  import { validateContactForm } from './contactValidation'
  import './contactForm.scss'

Constants (outside component):
  const SERVICE_ID  = 'service_kg75p6l'
  const TEMPLATE_ID = 'template_hkes3vq'

Component props: { onSuccess, onError }

State:
  const [fields, setFields]           = useState({ name: '', companyName: '', email: '', message: '' })
  const [errors, setErrors]           = useState({})
  const [sending, setSending]         = useState(false)
  const [submitStatus, setSubmitStatus] = useState('idle')

onChange handler (shared):
  function handleChange(e) {
    const { name, value } = e.target
    setFields(prev => ({ ...prev, [name]: value }))
    setErrors(prev => ({ ...prev, [name]: '' }))
  }

handleSubmit:
  1. const validationErrors = validateContactForm(fields)
  2. If Object.keys(validationErrors).length > 0 → setErrors(validationErrors), return
  3. setSending(true)
  4. emailjs.send(SERVICE_ID, TEMPLATE_ID, {
       from_name:    fields.name,
       company_name: fields.companyName,
       reply_to:     fields.email,
       message:      fields.message,
     }, import.meta.env.VITE_EMAILJS_PUBLIC_KEY)
     .then(() => {
       setSending(false)
       setSubmitStatus('success')
       if (onSuccess) onSuccess()
     })
     .catch(() => {
       setSending(false)
       setSubmitStatus('error')
       if (onError) onError()
     })

handleReset (for "Send another message"):
  setSubmitStatus('idle')
  setFields({ name: '', companyName: '', email: '', message: '' })
  setErrors({})

JSX conditional on submitStatus:

  'success':
    <div className="contact-form__success">
      <p>Message sent — we'll be in touch!</p>
      <button onClick={handleReset} className="contact-form__reset-btn">
        Send another message
      </button>
    </div>

  'error':
    Keep form visible + show:
    <p className="contact-form__error-msg">Something went wrong — please try again.</p>
    (form fields remain, user can resubmit)

  'idle' (default):
    Render all 4 fields + submit button

Field rendering pattern (repeat for each field):
  <div className="contact-form__field">
    <label className="contact-form__label" htmlFor="name">Name</label>
    <input
      className={`contact-form__input${errors.name ? ' contact-form__input--error' : ''}`}
      type="text"
      id="name"
      name="name"
      value={fields.name}
      onChange={handleChange}
      placeholder="Your name"
    />
    {errors.name && <span className="contact-form__field-error">{errors.name}</span>}
  </div>

Textarea for message uses same pattern but <textarea> tag instead of <input>.

NOTES & AFTERTHOUGHTS
======================

1. emailjs dependency
   --------------------
   @emailjs/browser should already be installed from EPIC-004.
   If not: npm install @emailjs/browser
   VITE_EMAILJS_PUBLIC_KEY must be set in .env — same key as EPIC-004.
   The template ID is different (template_hkes3vq) but the same service and
   public key apply. No .env changes needed if EPIC-004 is already set up.

2. EmailJS template variables to configure in dashboard
   -------------------------------------------------------
   The template_hkes3vq template in the EmailJS dashboard must contain:
     {{from_name}}    — sender's name
     {{company_name}} — sender's company (may be blank)
     {{reply_to}}     — sender's email (use as Reply-To in template settings)
     {{message}}      — message body
   Set the Reply-To field in the EmailJS template settings to {{reply_to}}
   so replies go directly to the customer's email address.

3. forwardRef and displayName
   ----------------------------
   When using forwardRef, always set ComponentName.displayName = 'ComponentName'
   for cleaner React DevTools debugging. Already noted in Feat-006 but worth
   flagging here too.

4. Modal inside NavBar tradeoff
   --------------------------------
   Placing modal state inside navBar.jsx is the simplest approach and avoids
   prop drilling through App.jsx → homePage.jsx → navBar.jsx. The tradeoff is
   that navBar.jsx imports a component from the components folder — this is an
   acceptable cross-folder import for this project scale. If the project grows
   and this feels messy, modal state can be lifted to App.jsx with a context
   or prop later.

5. contactForm.jsx reuse
   -----------------------
   contactForm.jsx is imported by both contactSection.jsx and getInTouchModal.jsx.
   This means any change to the form (new field, copy change, styling) only needs
   updating in one place. Keep the component prop-driven (onSuccess, onError) so
   both consumers can handle outcomes differently.

6. Scroll behaviour and hash links
   -----------------------------------
   The scrollIntoView approach is preferred over hash links (#contact) because
   it works cleanly with React Router and doesn't affect the URL bar.
   The id="contact" on the section element is kept as a fallback for any future
   anchor links but is not the primary scroll mechanism.

7. Routing cleanup risk
   ----------------------
   If /contact was linked anywhere outside navBar.jsx (e.g. footer, sitemap,
   other pages), those links will 404 after removal. Claude Code should search
   the codebase for '/contact' before deleting the route to catch any missed
   references.
